# دليل المساهمة في نظام الثورة الكمية

مرحباً بك في مشروع نظام الثورة الكمية! نحن نقدر مساهمتك ونريد أن نجعل عملية المساهمة سهلة وشفافة قدر الإمكان.

## قواعد المساهمة الأساسية

### 1. إرشادات الترميز (ملزمة)

#### المسافات البادئة
- **يجب** استخدام مسافتين (2) فقط للمسافة البادئة في جميع ملفات الكود
- تم تحديد هذه القاعدة في ملف `.editorconfig`
- أي كود لا يتبع هذه القاعدة سيتم رفضه

#### اصطلاحات التسمية

**المكونات (Components)**
- يجب استخدام PascalCase حصريًا
- أمثلة صحيحة: `DonationCard`, `UserProfile`, `QuantumSystem`
- أمثلة خاطئة: `donationCard`, `user_profile`, `quantum-system`

**المتغيرات والدوال (Variables & Functions)**
- يجب استخدام camelCase
- أمثلة صحيحة: `handleDonationSubmit`, `userData`, `currentUser`
- أمثلة خاطئة: `HandleDonationSubmit`, `user_data`, `current-user`

**ملفات CSS/Styling**
- يجب استخدام kebab-case لأسماء الفئات (classes)
- أمثلة صحيحة: `.donation-card-container`, `.user-profile-header`
- أمثلة خاطئة: `.donationCardContainer`, `.user_profile_header`

#### هيكل المكونات

**قاعدة مهمة**: يمنع منعًا باتًا كتابة Logic معقد داخل JSX مباشرة

✅ **الطريقة الصحيحة**:
```tsx
// استخراج المنطق إلى دالة منفصلة
const handleComplexOperation = useCallback(() => {
  // المنطق المعقد هنا
}, [dependencies]);

// في JSX
<button onClick={handleComplexOperation}>
  تنفيذ العملية
</button>
```

❌ **الطريقة الخاطئة**:
```tsx
// منطق معقد داخل JSX مباشرة
<button onClick={() => {
  // منطق معقد هنا - ممنوع!
  if (condition) {
    // عمليات معقدة
  }
}}>
  تنفيذ العملية
</button>
```

**قواعد إضافية**:
- يجب أن تكون المكونات صغيرة ومسؤولة عن وظيفة واحدة فقط
- استخدم `useEffect` و `useCallback` لتنظيم الكود
- فصل المنطق عن العرض

#### التعليقات

- **يجب** كتابة تعليقات واضحة ومفيدة باللغة العربية
- استخدم `// TODO:` لوضع علامة على المهام المستقبلية
- استخدم `// FIXME:` للإشارة إلى الأخطاء التي تحتاج إلى إصلاح

**أمثلة**:
```tsx
// دالة تسجيل الدخول - تتحقق من بيانات المستخدم وتحديث الحالة
const handleLogin = async (email: string, password: string) => {
  // TODO: إضافة التحقق من قوة كلمة المرور
  // FIXME: معالجة أفضل لأخطاء الشبكة
};
```

#### الأنماط (Styling)

- **يجب** استخدام Tailwind CSS بشكل أساسي
- **يجب** استخدام مكونات shadcn/ui وتعديلها عند الضرورة
- **يمنع** إعادة بناء المكونات من الصفر
- **يمنع** كتابة CSS مخصص إلا بعد موافقة قائد الفريق

### 2. قواعد المشروع العامة (ملزمة)

#### الواجهة (UI)
- **يجب** أن تكون الواجهة باللغة العربية
- **يجب** دعم RTL بشكل كامل
- أي تعديل لا يدعم RTL بشكل صحيح **سيتم رفضه**

#### الأداء
- **يجب** أن يكون الكود محسّنًا للأداء
- استخدم React Profiler لضمان سرعة التحميل
- تجنب الرندر غير الضروري للمكونات
- استخدم `React.memo`, `useMemo`, `useCallback` عند الحاجة

#### إمكانية الوصول (Accessibility)
- **يجب** الالتزام بمعايير إمكانية الوصول (WCAG) المستوى A كحد أدنى
- استخدم سمات `alt` للصور
- وفر تجربة مستخدم جيدة لذوي الاحتياجات الخاصة
- أي ميزة جديدة لا تتوافق مع هذه المعايير **سيتم رفضها**

## سير العمل (Git Workflow)

### استخدام Git Flow

**يجب** استخدام Git Flow بشكل صارم:

1. **الفروع الرئيسية**:
   - `main`: الإصدار المستقر
   - `develop`: فرع التطوير الرئيسي

2. **فروع الميزات**:
   - `feature/اسم-الميزة`: للميزات الجديدة
   - `hotfix/اسم-الإصلاح`: للإصلاحات العاجلة
   - `release/رقم-الإصدار`: لتحضير الإصدارات

### رسائل الـ Commits

**يجب** أن تكون رسائل الـ commits باللغة العربية وتتبع الصيغة التالية:

```
[نوع التغيير]: وصف مختصر للتغيير

وصف تفصيلي إضافي إذا لزم الأمر
```

**أنواع التغييرات**:
- `[Feature]`: إضافة ميزة جديدة
- `[Fix]`: إصلاح خطأ
- `[Style]`: تغييرات في التصميم
- `[Refactor]`: إعادة هيكلة الكود
- `[Docs]`: تحديث الوثائق
- `[Test]`: إضافة أو تعديل الاختبارات

**أمثلة**:
```
[Feature]: إضافة ميزة التشفير الكمي للملفات

- إضافة دالة encryptFile
- تحديث واجهة المستخدم
- إضافة اختبارات الوحدة
```

```
[Fix]: إصلاح مشكلة في تسجيل الدخول

كانت المشكلة في التحقق من صحة البريد الإلكتروني
```

### مراجعة الكود

- **يجب** أن تتم مراجعة كل تعديل (Merge Request) من قبل **ما لا يقل عن عضوين** من الفريق
- لا يمكن دمج أي تغيير بدون موافقة المراجعين
- يجب الرد على جميع التعليقات قبل الدمج

## عملية المساهمة

### 1. إعداد البيئة المحلية

```bash
# استنساخ المشروع
git clone [repository-url]
cd Top1

# تثبيت التبعيات
npm install

# تشغيل خادم التطوير
npm run dev
```

### 2. إنشاء فرع جديد

```bash
# التأكد من أنك في فرع develop
git checkout develop
git pull origin develop

# إنشاء فرع جديد للميزة
git checkout -b feature/اسم-الميزة-الجديدة
```

### 3. تطوير الميزة

- اتبع جميع إرشادات الترميز المذكورة أعلاه
- اكتب تعليقات واضحة باللغة العربية
- تأكد من دعم RTL
- اختبر الميزة بدقة

### 4. إرسال التغييرات

```bash
# إضافة التغييرات
git add .

# كتابة رسالة commit باللغة العربية
git commit -m "[Feature]: إضافة ميزة جديدة لإدارة المشاريع"

# رفع الفرع
git push origin feature/اسم-الميزة-الجديدة
```

### 5. إنشاء Pull Request

- اذهب إلى GitHub/GitLab
- أنشئ Pull Request من فرعك إلى `develop`
- اكتب وصفاً تفصيلياً للتغييرات
- أضف screenshots إذا كانت التغييرات تؤثر على الواجهة
- اطلب مراجعة من عضوين على الأقل

## الإبلاغ عن المشاكل

عند الإبلاغ عن مشكلة، يرجى تضمين:

1. **وصف المشكلة**: وصف واضح ومفصل
2. **خطوات إعادة الإنتاج**: كيفية إعادة إنتاج المشكلة
3. **السلوك المتوقع**: ما كان يجب أن يحدث
4. **السلوك الفعلي**: ما حدث بالفعل
5. **لقطات الشاشة**: إذا كانت مفيدة
6. **معلومات البيئة**: نظام التشغيل، المتصفح، إلخ

## طلب ميزات جديدة

عند طلب ميزة جديدة:

1. **وصف الميزة**: وصف واضح للميزة المطلوبة
2. **المبرر**: لماذا هذه الميزة مفيدة
3. **أمثلة الاستخدام**: كيف ستُستخدم الميزة
4. **البدائل**: هل فكرت في بدائل أخرى

## الحصول على المساعدة

إذا كنت بحاجة إلى مساعدة:

1. راجع الوثائق أولاً
2. ابحث في Issues الموجودة
3. أنشئ Issue جديد مع تفاصيل واضحة
4. استخدم التسميات المناسبة
